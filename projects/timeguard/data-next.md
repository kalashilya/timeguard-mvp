# Data next step

## Текущая версия

TimeGuard использует гибридную модель данных:

1. `localStorage` — быстрый frontend-сценарий планировщика;
2. Supabase — Auth, profiles, tasks, RLS, синхронизация и Cloud-чтение.

Такой подход позволяет сохранить стабильность учебного MVP и одновременно показать реальную базу данных.

## Что уже есть

- Supabase Auth;
- таблица `profiles`;
- таблица `tasks`;
- таблица `task_events`;
- RLS-политики;
- синхронизация локальных задач в Supabase;
- Cloud-страница для чтения задач из базы;
- уникальный `local_id` для связи локальной задачи и записи в базе.

## Следующий этап данных

Следующий этап — сделать основной Planner cloud-first.

Это значит:

- загружать задачи из Supabase сразу в `app.html`;
- сохранять каждое изменение напрямую в базу;
- поддержать offline-mode;
- решить конфликт версий между localStorage и Supabase;
- добавить realtime sync между вкладками и устройствами.

## Предлагаемые сущности production-версии

- `profiles` — профиль, email, роль, тариф;
- `tasks` — задачи пользователя;
- `task_events` — действия пользователя для аналитики;
- `subscriptions` — тарифный статус;
- `teams` — командные рабочие пространства;
- `team_members` — участники команд;
- `notifications` — напоминания;
- `calendar_integrations` — внешние календари.

## Правила доступа

- guest может смотреть лендинг;
- user может читать и редактировать только свои задачи;
- admin summary должен работать через backend function;
- тарифный статус должен проверяться на сервере;
- service-role key нельзя использовать во frontend.

## Почему не всё cloud-first уже сейчас

Цель текущего этапа — стабильный учебный MVP для демонстрации. Поэтому основной planner работает быстро и стабильно в браузере, а Supabase доказывает реальный auth, базу, RLS, синхронизацию и cloud-чтение.
