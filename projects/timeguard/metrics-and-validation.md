# Metrics and validation

## Гипотеза MVP

Если пользователь планирует день через TimeGuard, то он быстрее собирает реалистичное расписание и заранее видит пересечения по времени.

## Целевая гипотеза

TimeGuard должен сократить время ручной проверки плана дня с 10-15 минут до 3-5 минут и снизить количество незамеченных конфликтов времени.

## Целевая аудитория проверки

- студенты;
- молодые специалисты;
- менеджеры небольших команд;
- люди, которые совмещают учёбу, работу, встречи и личные задачи.

## Метрики успеха

| Метрика | Как измерять | Целевое значение |
|---|---|---|
| Время составления плана | Засечь время до и после TimeGuard | 3-5 минут на план дня |
| Количество найденных конфликтов | Сравнить ручной план и проверку TimeGuard | 80%+ конфликтов обнаружены до сохранения |
| Доля выполненных задач | Количество done / общее количество задач | 60%+ задач за день |
| Использование фильтров | Проверить, пользовался ли человек фильтрами | 50%+ тестовых пользователей |
| Использование экспорта | Нажатие TXT или JSON export | 30%+ тестовых пользователей |
| Понимание ценности | Ответ после теста | 8/10 и выше |
| Готовность пользоваться | Вопрос после демо | 60%+ ответов «да» |
| Готовность платить | Вопрос про Plus-тариф | 20%+ готовы рассмотреть оплату |

## План пилотной проверки

1. Найти 5-10 пользователей из целевой аудитории.
2. Попросить каждого составить план дня обычным способом.
3. Затем попросить составить тот же план через TimeGuard.
4. Засечь время.
5. Проверить, были ли пересечения задач.
6. Попросить пользователя оценить удобство по шкале от 1 до 10.
7. Собрать обратную связь по функциям: конфликт, прогресс, фильтры, экспорт, синхронизация.

## Вопросы для пользователя после теста

1. Понял ли ты, зачем нужен TimeGuard?
2. Помогла ли проверка конфликтов?
3. Удобно ли добавлять задачи?
4. Полезен ли прогресс выполнения?
5. Нужны ли фильтры по категории и приоритету?
6. Нужен ли экспорт плана?
7. Пользовался бы ты этим сервисом регулярно?
8. За какие функции ты был бы готов платить?

## Критерии успешного MVP

MVP можно считать успешным, если:

- пользователь понимает ценность за первые 30-60 секунд;
- задача добавляется без объяснений со стороны разработчика;
- конфликт времени понятен пользователю;
- пользователь может открыть кабинет и увидеть планы;
- Supabase-синхронизация проходит без ошибок;
- Cloud-страница показывает задачи из базы;
- пользователь может объяснить, чем TimeGuard лучше обычного todo-list.

## Вывод

Основная проверяемая ценность TimeGuard — не хранение задач, а помощь в создании реалистичного расписания. Поэтому главные метрики связаны со временем планирования, найденными конфликтами и ощущением контроля над днём.
