# Defense Q&A

## 1. В чём проблема проекта?

Обычные todo-листы помогают записывать задачи, но не проверяют, можно ли реально выполнить расписание. TimeGuard решает проблему пересечений по времени.

## 2. Кто пользователь?

Студенты, молодые специалисты и менеджеры, которым нужно совмещать учёбу, работу, встречи и личные дела.

## 3. В чём главная ценность?

Пользователь заранее видит конфликт и собирает реалистичный план дня, а не просто список задач.

## 4. Что является MVP?

MVP включает:

- создание профиля;
- добавление задач;
- проверку конфликтов;
- прогресс выполнения;
- фильтры;
- недельный обзор;
- экспорт;
- кабинет;
- тарифный сценарий;
- Supabase-синхронизацию;
- Cloud-страницу.

## 5. Что реально работает?

Реально работает:

- опубликованный сайт;
- frontend-логика планировщика;
- localStorage;
- Supabase Auth;
- таблицы Supabase;
- RLS;
- синхронизация задач;
- чтение задач обратно из Supabase.

## 6. Что является учебной имитацией?

Учебной имитацией являются:

- оплата тарифа;
- активация тарифа;
- admin-роль в интерфейсе;
- коммерческое ограничение Plus/Team.

## 7. Почему используется localStorage, если есть Supabase?

localStorage используется для стабильного frontend-сценария и мгновенной работы планировщика. Supabase добавлен как cloud-слой: регистрация, синхронизация и чтение задач из базы. Это нормальный подход для учебного MVP: сначала устойчивый frontend, затем cloud-слой.

## 8. Почему основной Planner не полностью cloud-first?

Потому что цель текущего MVP — показать ценность продукта и стабильный пользовательский путь. Полностью cloud-first архитектура требует более сложной двусторонней синхронизации, conflict resolution и backend-функций. Это следующий production-этап.

## 9. Зачем нужна Cloud-страница?

Она доказывает, что данные не просто лежат в браузере. После синхронизации задачи реально читаются из Supabase для текущего пользователя.

## 10. Как защищены данные?

Через Supabase RLS. Пользователь может читать и изменять только свои профиль и задачи. Во frontend используется только publishable key. Secret/service-role key не используется.

## 11. Почему admin-доступ не реализован через RLS?

В MVP admin-страница учебная. Настоящий admin-доступ лучше реализовывать через backend Edge Function или JWT claims, чтобы не создавать риск рекурсивных RLS-политик и не раскрывать агрегированные данные напрямую во frontend.

## 12. Почему нет Edge Functions?

Edge Functions описаны как production next step. Для учебного MVP достаточно frontend + Supabase RLS. В production туда нужно вынести проверку конфликтов, тарифы, платежи, admin summary и rate limiting.

## 13. Почему publishable key лежит во frontend?

Publishable key предназначен для frontend. Безопасность обеспечивается RLS. Опасно публиковать только secret или service-role key — он в проекте не используется.

## 14. Как проверяется конфликт задач?

При добавлении задачи приложение сравнивает новое время начала и окончания со всеми задачами выбранной даты. Если интервалы пересекаются, задача не сохраняется.

## 15. Как работает монетизация?

Free-план ограничивает количество задач и горизонт планирования. Plus и Team показаны как учебный тарифный сценарий. Реальные платежи требуют backend и payment webhooks.

## 16. Какие метрики успеха?

- время составления плана;
- количество найденных конфликтов;
- доля выполненных задач;
- использование фильтров;
- использование экспорта;
- понимание ценности;
- готовность пользоваться;
- готовность платить.

## 17. Что улучшить после MVP?

- cloud-first planner;
- realtime sync;
- Google Calendar integration;
- push-уведомления;
- backend Edge Functions;
- платежные webhooks;
- production admin panel;
- мобильная версия;
- командные планы.

## 18. Какой главный результат?

Проект показывает полный учебный путь: проблема, пользователь, MVP, интерфейс, логика, данные, Supabase, безопасность, монетизация и документация.

## 19. Чем проект сильнее обычного лендинга?

Это не просто страница. В проекте есть рабочий сценарий, сохранение данных, база Supabase, кабинет, cloud-чтение, экспорт, фильтры, метрики и тест-план.

## 20. Что сказать, если спросят про ограничения?

TimeGuard — учебный MVP. Основная цель — проверить ценность планирования без временных конфликтов. Оплата и admin являются имитацией, но Supabase Auth, база, RLS, синхронизация и cloud-чтение работают реально.
